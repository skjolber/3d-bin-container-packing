## Profile Snapshot - iter-001

### Analysis Method
Static code analysis (no runtime profiler available).

### Key Hot Paths Identified

#### DefaultPointCalculator2D.add(int, Placement) - called N times per benchmark sequence
1. Initial scan loop: O(endIndex) - already optimized with binary search for bounds
2. moveToXX/moveToYY handling: O(k log k) sort + O(k²) eclipses checks
3. constrainFloatingMax / constrainMax: O(n) per add
4. values.removeFlagged(): O(n) - two calls per add
5. **removeEclipsed(added): O(added × existingCount)** - inner loop with minX break
6. **values.sort(COMPARATOR_X_THEN_Y, endIndex): O(endIndex log endIndex)**
7. **while(endIndex < size && getMinX() <= xx): linear scan, O(k) worst case**
8. **updateIndexes(values): O(n) - unconditional iteration with conditional check**

#### DefaultPointCalculator3D.add(int, Placement) - called N times per benchmark sequence
1. Initial scan loop: O(endIndex)
2. moveToXX/YY/ZZ handling: O(k log k) sort + O(k²) eclipses checks
3. constrainMax / constrainFloatingMax: O(n) per add
4. **isEclipsed() called 10+ times per add, each O(otherValues.size())** - no early exit
5. **updateIndexes(otherValues): O(n) - conditional getIndex() check per iteration**

### Hotspot Ranking (estimated impact)

1. updateIndexes() in BOTH 2D and 3D:
   - Called once per add(), iterates ALL n points
   - Conditional: getIndex() call + compare + conditional branch
   - Fix: remove conditional, always call setIndex(i)
   - Estimated savings: ~2x reduction in loop body work

2. while loop after removeFlagged in 2D add():
   - Linear scan of sorted tail to extend endIndex beyond xx
   - Since tail is sorted, binary search gives O(log k) vs O(k)
   - Applies to Point2DFlagList.binarySearchPlusMinX()

3. isEclipsed() inner loop in 3D:
   - No early break for minX > point.minX
   - otherValues is NOT strictly sorted by minX (non-monotone due to addYY/addZZ inserts)
   - Cannot add safe break; continue is possible but adds a branch
   - Skipped as risky

### Data Structure Notes
- Point2DFlagList / Point3DFlagList: separate points[] and flag[] arrays (SoA layout)
  → Good cache behavior for points array access
  → Flag array is a boolean[] (1 byte each) - no bit packing
- Point2DFlagList.reset() iterates points.length (capacity) not size - wasteful but not in hot path
- Point3DFlagList.clear() uses loop with conditional - can use Arrays.fill

### Conclusion
Most promising optimization: remove conditional `if(p.getIndex() != i)` in updateIndexes()
  for BOTH 2D and 3D calculators. Additional gain: binary search for endIndex extension in 2D.
