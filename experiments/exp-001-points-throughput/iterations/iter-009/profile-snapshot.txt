## Code Analysis for iter-009

### 1. Where isEclipsed loops over otherValues
3D isEclipsed() (line 655-673):
  for (int index = 0; index < otherValues.size(); index++)
  -> otherValues.size() called on EVERY inner iteration
  -> point.getVolume() and point.getArea() also called EVERY inner iteration (not hoisted)

3D isEclipsedAtXX() (line 675-690):
  for (int index = otherValues.size() - 1; index >= 0; index--)
  -> otherValues.size() read once at loop init (ok)
  -> point.getVolume() and point.getArea() called on EVERY inner iteration (not hoisted)

### 2. Is otherValues.size() stable during isEclipsed() inner loop?
YES — within a single call to isEclipsed(), otherValues is never modified.
It IS modified in the outer loop (via otherValues.add(p)), but NOT within isEclipsed() itself.
Therefore: final int n = otherValues.size() before the loop is SAFE.

### 3. Are values/otherValues modified during the loops?
- Within isEclipsed(): otherValues.get() only reads, never writes size → safe to cache size
- Within isEclipsedAtXX(): same reasoning → safe to cache size

### 4. Where do moveX/moveY/moveZ create new objects?
3D (DefaultPointCalculator3D.java lines 323-330):
  added = p.moveX(xx);          // allocates new DefaultPoint3D
  added = p.moveX(xx, placement); // allocates new DefaultYZPlanePoint3D
Lines 365-371: p.moveY(yy), p.moveY(yy, placement)
Lines 411-417: p.moveZ(zz), p.moveZ(zz, placement)
Pre-checks (eclipsesMovedX/Y/Z) ALREADY EXIST in all three loops.

### 5. Is there a cheap condition to skip allocation?
The eclipsesMovedX/Y/Z checks against addedXX/YY/ZZ are already in place.
Checking against values[] would cost O(n) per candidate and is not worth it (analysis showed
allocation cost vs pre-check cost is unfavorable for typical problem sizes).

### Key Finding for iter-009
In isEclipsed(), three values are recomputed on every inner loop iteration:
  - point.getVolume()  (virtual call on polymorphic SimplePoint3D)
  - point.getArea()    (virtual call on polymorphic SimplePoint3D)
  - otherValues.size() (field read, but JIT may not hoist from field-based otherValues)

The JIT *might* already hoist these, but SimplePoint3D is polymorphic (5-6 concrete subclasses)
so the JIT may not fully devirtualize getVolume()/getArea() calls and therefore cannot hoist them.
Explicit local variable assignment guarantees hoisting regardless of JIT behavior.

Since isEclipsed() is called ~50 times per add() and the inner loop runs ~5-15 iterations,
total extra calls saved: ~50 × 10 × 2 = 1000 redundant method calls per add().

moveToYY.size() and moveToZZ.size() are NOT cached (inconsistent with moveToXX which IS cached).
While JIT likely hoists these, making it explicit is safe and consistent.
