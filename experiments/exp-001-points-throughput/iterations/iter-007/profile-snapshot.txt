## Analysis of DefaultPointCalculator2D.add()

### Q1: How many times is removeFlagged() called per add() invocation?
**Twice.**

### Q2: Lines and purpose of each call
- **Line 437**: `endIndex -= values.removeFlagged()`
  Purpose: Remove items flagged by the constraint operations (constrainMaxX/Y etc.)
  and by the main processing loop (line 285: `values.flag(i)` for swallowed points,
  line 154: `values.flag(index)` for the source point consumed by the placement).
  All flagged items here are in positions [0..endIndex_initial-1] (confirmed by code
  inspection — all constrain loops iterate [pointIndex..endIndex]).

- **Line 455**: `endIndex += added - values.removeFlagged()`
  Purpose: Remove items flagged by removeEclipsed() (new projected points that are
  eclipsed by existing sorted points). Returns G = count of flagged new items.

### Q3: What does endIndex represent and how is it adjusted?
- `endIndex` = exclusive upper bound index: first position where minX >= xx
  (where xx = placement.getAbsoluteEndX() + 1)
- Initialized at line 226: `values.binarySearchPlusMinX(index + 1, placement.getAbsoluteEndX())`
- After first removeFlagged (line 437): `endIndex -= F` where F = items removed
  (All F items are in [0..endIndex_initial-1], confirmed by loop ranges)
- After values.move(added): logical shift, endIndex not yet bumped
- After second removeFlagged (line 455): `endIndex += added - G`
  The `added` new items all have minX < xx by construction.
  Net result: endIndex = endIndex_initial + added - F - G ✓

### Q4: Does Point2DFlagList have binarySearchPlusMinX()?
YES — lines 247-249 of Point2DFlagList.java:
  `public int binarySearchPlusMinX(int minIndex, int key)` — returns exclusive result.
However, we do NOT need to use it for endIndex recomputation in the merged approach.
The arithmetic formula `endIndex += added - values.removeFlagged()` already works
correctly (see correctness proof in plan.md).

### Q5: Is restoring isFlag check in removeEclipsed inner loop safe when deferring removeFlagged?
YES — and it is REQUIRED.

When we defer the first removeFlagged(), original flagged items remain in positions
[added..size-1] during removeEclipsed(added). Without the isFlag check, a new item
could be compared against a flagged (doomed) original item and incorrectly marked as
eclipsed by an item that doesn't actually exist in the final set.

Adding back `if(values.isFlag(index)) continue;` in the inner loop correctly skips
flagged originals. Since in the typical case only a small fraction of items are flagged,
this check is almost always branch-predicted correctly (false branch).

The outer loop already has `if(values.isFlag(i)) continue;` — no change needed there
(new items are always unflagged by setAll/move).
